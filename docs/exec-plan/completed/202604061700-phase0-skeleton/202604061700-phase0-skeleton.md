## 三、与 plan 的偏离

### 偏离 1：运行时依赖未常驻 `go.mod`

- **原计划**：T2 通过 `go get` 引入 grpc / protobuf / fsnotify / viper / cobra / testify / backoff/v4 / x/sync / doublestar/v4 / x/time / hanwen/go-fuse/v2 共 11 个三方库到 `go.mod`。
- **实际**：执行 `go mod tidy` 后，除 proto 工具链（由 `tools/tools.go` 锁定）之外的所有运行时依赖被剪掉，因为 Phase 0 没有任何 `.go` 源文件实际 import 它们。
- **原因**：保留这些依赖需要引入"假依赖哨兵文件"，违反 CLAUDE.md "保持简单" 原则。
- **影响**：零负面影响。已通过 `go get` 验证 11 个库均可在 Go 1.25.2 下正常下载（Go module 缓存已就绪）。Phase 1 各 `pkg/` 子包真正 import 时会自然回到 `go.mod`。
- **`go.mod` 终态**：
  - `require google.golang.org/grpc/cmd/protoc-gen-go-grpc v1.6.1`
  - `require google.golang.org/protobuf v1.36.11`
  - `require google.golang.org/grpc v1.80.0` // indirect

### 偏离 2：`make tools` 目标小范围扩展

- **原计划**：T3 仅修改 Makefile 的 `proto` 目标。
- **实际**：同时在 `tools` 目标追加 `go install` proto 插件二进制的步骤，让 `make proto` 能端到端跑通。
- **原因**：proto 插件可执行文件必须落到 `~/go/bin/`，否则 protoc 找不到它们；版本号已由 `tools/tools.go` 经 `go.mod` 锁定，扩展不引入重复硬编码。
- **影响**：使本机首次跑 `make proto` 即可成功；不影响 CI（`make proto` 在无 `.proto` 时直接 exit 0 兜底）。

### 偏离 3：未引入"假依赖哨兵文件"

- **原计划隐含**：T2 应保证所有运行时依赖出现在 `go.mod` 的 require 段。
- **实际**：放弃这一硬性要求（见偏离 1），改用"Phase 1 自然引入"策略。
- **原因**：哨兵文件违反 CLAUDE.md 第 3 条 "Implementation Guidelines"——不要为假想需求做抽象/占位。
