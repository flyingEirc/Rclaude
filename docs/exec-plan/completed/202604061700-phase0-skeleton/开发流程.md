- 拉取依赖：先 `go get` 全部产线 + 测试依赖（grpc / protobuf / fsnotify / viper / cobra / testify / backoff/v4 / x/sync / doublestar/v4 / x/time / hanwen/go-fuse/v2），证明全部可在 Go 1.25.2 下解析与下载
- `go.mod` 最终内容（`go mod tidy` 后）：
  - require `google.golang.org/grpc/cmd/protoc-gen-go-grpc v1.6.1`
  - require `google.golang.org/protobuf v1.36.11`（提供 protoc-gen-go）
  - require `google.golang.org/grpc v1.80.0` // indirect
- 验证命令与结果：
  - `go build ./...` → exit 0
  - `go mod tidy` → exit 0
  - `golangci-lint run ./...` → `0 issues.`
  - `go test -race -count=1 -timeout 120s ./...` → 全部 `[no test files]` exit 0
- **对原计划的偏离与说明**（必读）：
  1. **运行时依赖未常驻 go.mod**：原计划列出的 grpc / viper / cobra / fsnotify / testify / backoff/v4 / doublestar/v4 / x/sync / x/time / go-fuse/v2 等被 `go mod tidy` 剪掉了，因为 Phase 0 没有任何 Go 源文件实际 import 它们。已通过 `go get` 验证它们均能在 Go 1.25.2 下正常下载（go module 缓存已就绪），Phase 1 各 `pkg/` 子包真正 import 时会自然回到 `go.mod`。**未引入"假依赖哨兵文件"**（违反 CLAUDE.md "保持简单" 原则）。proto 工具链通过 `tools/tools.go` 锁定，是 Phase 0 唯一必须常驻 go.mod 的依赖。
  2. **`go build -tags=tools ./...` 会失败**：因为 `protoc-gen-go` / `protoc-gen-go-grpc` 是 `main` 包，按 Go 规范不允许被任何文件 `import`。tools.go 模式只用来让 `go mod tidy` 把工具模块版本写入 `go.mod`，**不用于编译**。要安装可执行二进制需用 `go install google.golang.org/protobuf/cmd/protoc-gen-go`（带 `@version` 或在模块内省略版本）。这是社区惯例，不是 bug。
